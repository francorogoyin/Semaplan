Plan de implementacion para API de IA de Semaplan.

Fecha de redaccion inicial: 2026-04-16.
Fecha de actualizacion: 2026-04-27.

Objetivo.

Construir una API de solo lectura para que GPT y Claude puedan
consultar datos de Semaplan con autorizacion explicita del usuario.

La API debe cubrir el estado operativo real de la app:

- agenda y eventos;
- tareas;
- habitos;
- slots vacios, tipos de slot y planes de slot;
- planes semanales;
- planes por periodos;
- Archivero;
- Baul;
- metas;
- contexto compacto.

La API debe exponer vistas legibles y seguras, sin devolver el blob
crudo completo de `public.estado_usuario.estado`.

Este documento ya no describe solo la idea general. Debe servir como
guia lista para implementar: orden de fases, contrato publico,
normalizadores, tests y criterio de cierre.

## Decision tecnica

Crear un Semaplan AI Gateway como capa unica de lectura.

- Backend unico:
  `supabase/functions/semaplan-ai/index.ts`
- Consumo de GPT:
  GPT Actions + OpenAPI
- Consumo de Claude:
  MCP remoto o tool use intermedio

No crear dos backends separados para GPT y Claude.

## Fuente real de datos

La fuente remota de verdad es `public.estado_usuario`.

La columna `estado` contiene el blob JSON completo del usuario.

La app operativa real hoy vive principalmente en `login.html`.
`index.html` es landing publica y `Semaplan.html` redirige a
`login.html`.

Antes de codificar un normalizador, relevar en `login.html` la
estructura actual del subsistema correspondiente.

Claves persistidas relevantes observadas hoy:

- `Objetivos`
- `Eventos`
- `Metas`
- `Slots_Muertos`

- `Planes_Periodo`
- `Plantillas_Subobjetivos`
- `Tareas`
- `Tareas_Cajones_Definidos`

- `Habitos_Registros`
- `Baul_Objetivos`
- `Archiveros`

Si durante la implementacion aparece una clave adicional que sea

## Principios de version 1

La version 1 debe ser estrictamente read-only.

Permitido:

- leer estado normalizado por dominios;
- consultar por fecha, rango, cajon, estado o texto;
- devolver resumenes compactos para IA.

No permitido:

- crear, editar, mover o borrar datos;
- disparar sync;
- devolver el blob completo;
- devolver credenciales, tokens, emails de terceros o datos de debug;
- usar cookies;
- exponer service role key al frontend.

## Autenticacion

La integracion inicial debe usar tokens de lectura de IA por usuario.

Tabla nueva recomendada:
`public.tokens_ia_usuario`

Columnas recomendadas:

- `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
- `usuario_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE`
- `nombre TEXT NOT NULL DEFAULT 'Integracion IA'`
- `token_hash TEXT NOT NULL UNIQUE`
- `scopes TEXT[] NOT NULL DEFAULT ARRAY['read']`
- `ultimo_uso_en TIMESTAMPTZ`
- `creado_en TIMESTAMPTZ NOT NULL DEFAULT NOW()`
- `revocado_en TIMESTAMPTZ`

Reglas:

- El token real se muestra una sola vez.
- La base guarda solo hash SHA-256.
- El cliente envia `X-Semaplan-AI-Token`.
- La Edge Function hashea el token entrante y busca su hash.
- Si el token no existe o esta revocado, rechazar.
- Leer solo el estado del `usuario_id` asociado al token.
- Aceptar `Authorization: Bearer <jwt>` solo como modo interno de
  prueba para la app web.

## Base URL

Base publica:
`https://<project>.supabase.co/functions/v1/semaplan-ai`

Todos los endpoints deben vivir bajo la misma Edge Function.

## Reglas de normalizacion por dominio

### Agenda

- inspeccionar `login.html` para mapear eventos reales;
- horarios en `America/Buenos_Aires`;
- omitir eventos sin fecha valida;
- incluir nombre de objetivo asociado si existe;
- incluir nota solo si es breve;
- si una nota supera 1000 caracteres, truncar y marcar
  `Texto_Truncado: true`.

### Tareas

- no mezclar tareas con eventos aunque puedan estar vinculadas;
- si la tarea esta pospuesta, devolver fecha y hora vacias;
- exponer solo referencias minimas a slot, plan o evento;
- no exponer borradores ni flags solo de UI.

### Habitos

- devolver solo habitos visibles para fecha y modo pedido;
- incluir programacion y meta en formato legible;
- resumir registros relevantes sin bajar historial entero pesado;
- no exponer filtros ni estados internos de UI.

### Slots

- incluir `Fecha`, `Hora`, `Tipo_Id`, `Tipo_Nombre`,
  `Titulo_Visible` y plan explicito si existe;
- si un slot tiene items planeados, devolver lista corta legible;
- no duplicar metadata pesada por cada entrada si puede resumirse.

### Planes semanales

- exponer snapshot base y diff resumido;
- no bajar toda la semana completa dentro de `/contexto`;
- si la semana no existe, devolver estructura vacia consistente.

### Planes por periodos

- devolver periodos, objetivos, subobjetivos y partes con:
  `Id`, `Nombre`, `Estado`, `Progreso`, `Unidad`,
  fechas y relaciones padre-hijo;
- resumir avances;
- no exponer configuracion `UI` salvo lo minimo necesario.

### Archivero

- ordenar notas por fecha mas nueva primero;
- buscar por texto normalizado;
- si no hay titulo, `Titulo: null`;
- si no hay etiquetas, `Etiquetas: []`;
- no devolver color de fondo salvo necesidad concreta;
- si hay adjuntos, no devolver base64 completo en v1.

### Baul

- devolver objetivos accionables y archivados visibles;
- incluir categoria legible;
- incluir timeline si existe;
- resumir metadatos visibles;
- no devolver estructuras duplicadas pesadas.

### Metas

- devolver nombre, periodo, unidad, objetivo, progreso y estado;
- si no se puede inferir progreso con seguridad, usar `Detalle`;
- agregar test antes de publicar si la estructura real queda ambigua.

## CORS, errores y limites

CORS:

- responder siempre `OPTIONS`;
- permitir `Authorization`, `apikey`, `content-type` y
  `X-Semaplan-AI-Token`;
- usar `Access-Control-Allow-Origin: *` solo sin cookies.

Formato de error:

```json
{
  "Ok": false,
  "Error": "No autorizado",
  "Detalle": "Token invalido."
}
```

Codigos HTTP:

- `200` exito
- `400` parametros invalidos
- `401` sin token o JWT invalido
- `403` token revocado o scope insuficiente
- `404` ruta inexistente
- `429` rate limit
- `500` error inesperado

Rate limit minimo de v1:

- 60 requests por minuto por token
- 500 requests por dia por token

Si no entra tabla de rate limit en el primer commit tecnico, dejar el
codigo preparado y documentar deuda explicita. No cerrar la API
publica sin rate limit.

## Fases de implementacion

### Fase 0. Cerrar contrato

Objetivo:

la funcion.

Salida esperada:

- este documento cerrado;
- OpenAPI bosquejado;

### Fase 1. Autenticacion y esqueleto

Archivos:

- migration `tokens_ia_usuario`
- Edge Function con `OPTIONS`, `/salud` y validacion de token

- `/salud` responde `200`

`Agregar autenticacion base de semaplan-ai`

### Fase 2. Lectura segura del estado

Objetivo:
leer `estado_usuario` del usuario autenticado y filtrar campos seguros.

Tests minimos:

- no mezclar usuarios
- no devolver blob crudo completo
- no exponer tokens ni credenciales

Commit recomendado:
`Agregar lectura segura de estado para semaplan-ai`

### Fase 3. Contexto y agenda

`Agregar contexto y agenda a semaplan-ai`

### Fase 4. Operacion diaria

## Matriz minima de pruebas por endpoint

- `/salud`: responde `200` y version
- `/contexto`: respeta rango y no devuelve payload excesivo
- `/agenda`: valida rango y omite eventos invalidos
- `/tareas`: filtra tareas y no mezcla objetivos
- `/habitos`: calcula estado visible segun fecha y modo
- `/slots`: devuelve plan de slot y tipo visible
- `/planes/semana`: devuelve snapshot y diff
- `/planes/periodos`: devuelve jerarquia segura
- `/archivero`: ordena y no baja base64 de adjuntos
- `/archivero/buscar`: normaliza acentos
- `/baul`: devuelve categoria y timeline si existe
- `/metas`: devuelve progreso o detalle seguro
- `/openapi.json`: refleja endpoints y auth real

## Integracion con GPT

Crear Custom GPT `Semaplan`.

Usar GPT Actions con:

- OpenAPI de
  `Documentacion/Planes/OpenAPI_Semaplan_AI.yaml`
- auth por API key en `X-Semaplan-AI-Token`

Instrucciones base del GPT:

- responder solo cuando el usuario pida datos de Semaplan;
- no inventar horarios, notas, tareas, habitos ni objetivos;
- pedir rango de fechas si la pregunta temporal es ambigua;
- si el endpoint devuelve vacio, decirlo;
- no modificar datos en v1.

## Seguridad

- no usar endpoints sin autenticacion;
- no loguear contenido de notas, agenda, tareas ni habitos;
- no devolver stack traces al cliente;
- no aceptar queries de busqueda > 200 caracteres;
- no aceptar rangos temporales abiertos o ilimitados;
- no mezclar datos entre usuarios;
- no guardar tokens reales en base ni localStorage.

## Escritura futura

La escritura queda fuera de v1.

Posibles scopes futuros:

- `read`
- `write_notes`
- `write_tasks`

- `write_agenda`

Si algun dia entra escritura, debe requerir:

- auditoria;

## Cierre de tarea

La tarea se considera cerrada cuando existan:

- API read-only segura;
- tokens de lectura por usuario;
- OpenAPI usable por GPT;
- camino documentado o implementado para Claude;
- pruebas reportadas;
- deploy ejecutado o comando pendiente explicitado.
