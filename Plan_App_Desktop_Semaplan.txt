Plan para una app desktop de Semaplan

Objetivo

Evaluar y dejar planificado como construir una app desktop
practicamente identica a la web actual, conectada al mismo
backend y con el menor re-trabajo posible.

Respuesta corta

No es tanto quilombo si el objetivo es:

- misma UI;
- mismo backend;
- misma cuenta;
- app instalable para Windows o Mac.

Si ademas queres que "se conecte automaticamente" a la
sesion que ya existe en la web, ahi sube la complejidad, pero
sigue siendo viable si se disena bien.

------------------------------------------------------------
1. Que significa "desktop identica"
------------------------------------------------------------

Hay dos caminos muy distintos:

Camino A. Wrapper de la webapp

La app desktop es basicamente un contenedor nativo que corre
la misma app web por dentro.

Ventajas

- reutilizas casi todo el codigo actual;
- no rehaces la UI;
- el mantenimiento es mucho mas barato;
- el comportamiento queda casi igual en web y desktop.

Desventajas

- seguis atado a una arquitectura web;
- algunas cosas nativas requieren puentes extra;
- no compartis sesion automaticamente con el navegador.

Camino B. App desktop realmente separada

Rehacer frontend o parte del frontend como app nativa o con
otro stack.

Ventajas

- control total sobre comportamiento desktop;
- mejor integracion nativa;
- posible mejor rendimiento en algunas zonas.

Desventajas

- muchisimo mas trabajo;
- duplicas logica y mantenimiento;
- mas riesgo de divergencia entre web y desktop.

Recomendacion

Para Semaplan, el camino correcto es claramente el A.
Primero wrapper. Despues, si queres, funciones nativas
encima.

------------------------------------------------------------
2. Tecnologias posibles
------------------------------------------------------------

Opcion 1. Electron

Es la opcion mas directa y flexible.

Pros

- enorme ecosistema;
- facil para empaquetar;
- simple para abrir la misma app web o local;
- buen soporte para auto-update, deep links, notificaciones,
  menu del sistema, archivos.

Contras

- mas pesado en RAM y disco;
- packaging mas grande;
- hay que ser prolijo con seguridad.

Opcion 2. Tauri

Mas moderna y liviana.

Pros

- binarios chicos;
- mejor uso de recursos;
- muy buena opcion si queres algo mas fino.

Contras

- curva un poco mas tecnica;
- integra mejor si mas adelante aceptas un poco de trabajo
  adicional de setup;
- debugging y packaging pueden ser menos directos si no lo
  venis usando.

Recomendacion practica

Si queres llegar rapido:

- Electron.

Si queres algo mas prolijo y liviano, y no te molesta un poco
mas de setup:

- Tauri.

Como tu app actual es un solo index.html sin build, Electron
te va a doler menos al principio.

------------------------------------------------------------
3. Version minima viable recomendada
------------------------------------------------------------

Objetivo de la V1

Tener una app instalable que:

- abra Semaplan;
- use el mismo backend Supabase;
- permita login normal;
- funcione como la web;
- tenga icono, instalador y ventana propia.

Importante

En la V1 no intentaria compartir sesion con la web.
Haria login propio dentro de la desktop.

Por que

Porque eso reduce mucho la complejidad inicial y te deja ver
si realmente queres mantener una desktop antes de meterte en
handoff de auth.

------------------------------------------------------------
4. Arquitectura sugerida
------------------------------------------------------------

Base de la app

- Un proceso desktop minimo.
- Una ventana principal.
- La UI actual de Semaplan cargada dentro.

Dos variantes posibles

Variante 1. Cargar semaplan.com dentro de la app desktop

La app desktop abre la version hospedada.

Pros

- casi cero duplicacion;
- deploys web impactan tambien en desktop;
- mantenimiento minimo.

Contras

- dependes totalmente de la web publicada;
- si rompes produccion, rompes desktop tambien;
- menos control sobre versiones.

Variante 2. Empaquetar la UI dentro de la app desktop

La app instala una copia del frontend y habla con el mismo
Supabase.

Pros

- mas control de version;
- podes sacar releases desktop separadas;
- produccion web y desktop no quedan siempre acopladas.

Contras

- build y release algo mas complejos;
- hay que pensar actualizaciones.

Recomendacion

Si lo queres serio, mejor Variante 2.
Si lo queres probar rapido, Variante 1 para validar interes.

------------------------------------------------------------
5. El verdadero tema: autenticacion
------------------------------------------------------------

Esto es lo importante.

Desktop y navegador web no comparten automaticamente sesion.
Aunque el usuario ya este logueado en Chrome, la app desktop
no va a heredar sola ese estado.

Escenarios posibles

Escenario 1. Login separado en desktop

La desktop muestra la misma pantalla de login y el usuario
entra ahi tambien.

Ventajas

- simple;
- seguro;
- facil de mantener;
- sin magia rara.

Desventajas

- el usuario tiene que loguearse una vez mas.

Mi recomendacion para V1

Este.

Escenario 2. Handoff de sesion desde la web

La web ofrece algo como:

"Abrir en app desktop"

Y le pasa a la desktop un token temporal de una sola vez.

Flujo sugerido

1. Usuario logueado en web toca "Abrir en desktop".
2. La web pide al backend un token corto de enlace.
3. La desktop se abre por deep link.
4. La desktop canjea ese token por sesion.
5. El token expira o se invalida.

Ventajas

- experiencia muy buena;
- parece login automatico;
- no comparte credenciales de forma burda.

Desventajas

- requiere deep linking;
- requiere una tabla o flujo de tokens temporales;
- requiere mas hardening de seguridad;
- requiere mas testeo.

Escenario 3. Compartir cookies o sesion del navegador

No lo recomiendo.

Es fragil, dependiente del sistema y mas sucio de mantener.

------------------------------------------------------------
6. Fases recomendadas
------------------------------------------------------------

Fase 1. Prototipo desktop minimo

Objetivo

Tener Semaplan abriendo como app de escritorio.

Tareas

- crear shell Electron o Tauri;
- abrir una ventana principal;
- cargar la app web o una copia local;
- definir icono y nombre;
- probar login manual;
- validar sync con Supabase.

Criterio de cierre

La app abre, loguea, guarda y cierra sin diferencias graves.

Fase 2. Empaquetado e instalador

Objetivo

Que no sea solo un experimento de desarrollo.

Tareas

- generar instalador Windows;
- definir app id;
- definir iconos;
- revisar permisos;
- probar en otra PC limpia.

Criterio de cierre

La app se instala y desinstala bien.

Fase 3. Extras nativos utiles

Objetivo

Que empiece a sentirse desktop de verdad.

Opciones utiles

- abrir al iniciar sistema;
- notificaciones nativas;
- atajos globales;
- export local rapido;
- abrir archivos de backup;
- deep links.

Criterio de cierre

Hay beneficios concretos frente a usar la web en una pestana.

Fase 4. Handoff desde la web

Objetivo

Abrir la desktop desde la web con sesion resuelta.

Tareas

- definir protocolo tipo semaplan://
- crear token temporal de enlace
- agregar endpoint de canje
- abrir app desde la web
- invalidar token luego del uso
- registrar expiracion y abuso

Criterio de cierre

El usuario puede pasar de web a desktop sin volver a poner
mail y password, pero sin comprometer seguridad.

------------------------------------------------------------
7. Riesgos reales
------------------------------------------------------------

Riesgo 1. Duplicar demasiado pronto

Si haces una desktop separada de verdad, te metes en una
segunda app. Hay que evitar eso.

Riesgo 2. Mezclar sesiones de forma insegura

Si improvisas el login automatico, podes abrir agujeros
feos. Los tokens temporales tienen que ser:

- de un solo uso;
- cortos;
- expirables;
- auditables.

Riesgo 3. Desktop y web divergen

Si una usa un frontend empaquetado y otra la ultima version
web, podes empezar a tener bugs distintos.

Riesgo 4. Falta de testing real

Una desktop puede verse "igual", pero cambiar cosas:

- manejo de foco;
- copiar/pegar;
- ventanas modales;
- abrir links;
- notificaciones;
- drag and drop.

------------------------------------------------------------
8. Cosas que si valen la pena en desktop
------------------------------------------------------------

Estas si justifican una version desktop:

- abrir mas rapido que una pestana;
- quedarse fija como herramienta principal;
- notificaciones nativas;
- mejor integracion con archivos;
- modo foco mas "app" que "sitio";
- menos dependencia del navegador del usuario.

Estas no justifican por si solas:

- "que se vea como app";
- "tener iconito";
- "envolver la web porque si".

------------------------------------------------------------

Si yo tuviera que hacerlo en Semaplan, haria esto:

Paso 1

Electron minimo.
Login propio.
Misma app.
Mismo Supabase.

Paso 2

Instalador Windows.
Icono.
Pruebas en otra PC.

Paso 3

Uno o dos extras nativos:

- abrir al iniciar;
- notificaciones;

Paso 4

------------------------------------------------------------
10. Plan ejecutable
------------------------------------------------------------

Si queres construirla de forma ordenada, este seria un plan
razonable.

Semana 1. Prototipo

- crear proyecto Electron o Tauri;
- abrir Semaplan dentro;
- validar login;
- validar sync;
- validar que drag/drop y modales no exploten.

Entregable

App desktop corriendo en tu maquina.

Semana 2. Packaging

- generar instalador;
- resolver iconos;
- revisar comportamiento al abrir links;
- probar en otra PC.

Entregable

Primera build instalable.

Semana 3. Hardening desktop

- manejo de errores;
- logs;
- notificaciones;
- abrir al iniciar;
- revisar seguridad basica.

Entregable

Version usable por vos todos los dias.

Semana 4. Handoff web -> desktop

- definir deep link;
- crear token temporal;
- canje de sesion;
- invalidacion;
- pruebas de expiracion.

Entregable

Experiencia fluida entre web y desktop.

------------------------------------------------------------
11. Criterio para decidir si hacerlo ya o no
------------------------------------------------------------

Yo lo haria ya si:

- usas Semaplan muchas horas al dia;
- queres notificaciones nativas;
- te molesta depender del navegador;
- queres posicionarla como herramienta principal.

Yo lo patearia si:

- todavia estas estabilizando web, sync y auth;
- no tenes aun smoke tests de produccion;
- la prioridad sigue siendo confiabilidad web.

------------------------------------------------------------
12. Conclusión
------------------------------------------------------------

Una app desktop identica a Semaplan web es totalmente viable
y no deberia ser un proyecto monstruoso si la haces como
wrapper.

La parte facil es la UI.
La parte que hay que disenar bien es la autenticacion.

La forma mas sana de hacerlo es:

- V1 con login propio dentro de desktop;
- V2 con handoff desde web por token temporal si mas adelante
  realmente lo queres.

Eso te deja avanzar rapido sin meterte de entrada en la parte
mas delicada.
